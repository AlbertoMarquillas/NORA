# 7. Action and Expression

#definition: layer responsible for executing actions and expressing the system's internal decisions toward the physical world, digital systems, or human users.

The Action and Expression module represents the operational output layer of NORA. While perception modules capture information from the environment and the cognitive core decides what should happen, this module is responsible for actually performing those decisions.

In architectural terms, this layer converts intentions, plans, and decisions produced by the cognitive core and planning system into concrete effects that can be perceived externally.

These effects may occur in several domains:

* human communication (speech, visual feedback, interaction signals)
* robot physical behavior (movement, posture, LEDs)
* digital actions (sending messages, triggering services)
* environmental control (IoT or smart devices)

This module therefore acts as the final execution layer of the system.

The separation between "planning" and "action" is fundamental to maintain modularity. The planning system determines *what* should be done, while the action layer determines *how* the system performs it.

For example:

User intention: "turn on the lights"

Planner output:

1. identify the device
2. verify permissions
3. trigger home automation command

Action layer execution:

1. send command to Home Assistant
2. update system state
3. express confirmation to the user

This architecture allows NORA to support multiple forms of expression simultaneously. A single decision from the planner may produce multiple coordinated outputs such as:

* speaking a response
* displaying information on screen
* changing LED colors
* moving the robot head

These multimodal outputs allow the system to appear responsive, expressive, and understandable to human users.

The Action and Expression module also ensures that outputs are synchronized and consistent with the current system state. For example, speech output should not overlap with listening states, and mechanical movements should respect safety constraints.

Because NORA operates both as a conversational agent and as a physical embodied system, this layer is responsible for bridging the cognitive layer with the physical embodiment of the robot.

Typical responsibilities of this module include:

* executing speech output
* generating visual output on the robot display
* controlling LEDs and visual indicators
* moving mechanical components
* activating sensors intentionally
* controlling external smart devices
* sending notifications to remote systems

This layer does not decide *why* an action should occur. That responsibility belongs to the planning and agent system. Instead, this layer focuses on **reliable execution, synchronization, and safe interaction with the environment**.

The module is divided into several specialized submodules responsible for different output channels, including speech, multimedia, visual interfaces, physical movement, camera usage, IoT integration, and external communication.

Each submodule exposes standardized action primitives that can be triggered by the planner or specialized agents.

These primitives form the fundamental building blocks of the robot's observable behavior.

In summary, the Action and Expression module is the layer where NORA's decisions become real-world effects.

# 7.1 Voice Output

#definition: subsystem responsible for generating spoken responses and verbal communication from NORA to human users.

The Voice Output subsystem provides the primary natural communication channel from the robot toward humans. It allows NORA to transform textual responses, instructions, explanations, or feedback into audible speech that users can understand without needing to look at a screen.

Speech output is one of the most important expressive modalities in human-robot interaction because spoken language is the most intuitive way for humans to receive information, confirmations, and explanations from an intelligent system.

Within the overall architecture, voice output typically receives text generated by the dialogue system, planning system, or specialized agents. That text is then processed and synthesized into audio through a text-to-speech engine.

This subsystem therefore performs several responsibilities:

* transforming text responses into speech
* controlling voice characteristics
* managing audio playback
* synchronizing speech with other robot actions

The voice output pipeline normally follows these stages:

1. text generation from dialogue or agents
2. optional speech markup enrichment
3. voice synthesis
4. audio streaming or playback

This pipeline ensures that NORA can communicate in a clear and expressive way while remaining consistent with the system's emotional and contextual state.

Because NORA may interact with multiple users, the subsystem supports personalization features such as different voices per user, different languages, and contextual speech styles.

Voice output may also adapt to the robot's emotional state or the type of interaction. For example, an alert message may use a more direct tone, while educational explanations may be slower and more articulated.

The subsystem also integrates with the system state machine to ensure safe and coherent interaction. For example:

* speech should pause if the system transitions to a listening state
* urgent alerts may interrupt ongoing speech
* confirmations should occur immediately after certain actions

These coordination rules ensure that voice output remains synchronized with the rest of the robot's behavior.

### Text-to-Speech Engine (TTS)

#definition: engine responsible for converting textual content into synthesized speech.

The TTS engine generates an audio waveform from a textual input. This component may use either local models or external services depending on the system configuration.

Typical capabilities include:

* high quality neural voice synthesis
* multi-language support
* configurable voice profiles
* real-time generation

The TTS engine outputs an audio stream that is then played through the robot speakers.

#definition: speech synthesis markup that controls pronunciation and expressive characteristics of the generated speech.

Speech Synthesis Markup Language (SSML) allows the dialogue system to influence how the sentence is spoken rather than only what is spoken.

SSML directives may include:

* pauses
* emphasis
* phonetic corrections
* prosody adjustments
* speech rate modifications

This allows NORA to produce more natural sounding speech and clearer explanations.

---

### Voice Characteristics

#definition: configurable parameters that define the acoustic identity of NORA's voice.

Several properties can be adjusted to adapt speech to context or user preferences:

* voice identity
* pitch
* speech rate
* emotional tone
* pronunciation dictionary

Different users may have personalized voice settings stored in their user profiles.

---

### Language Adaptation

#definition: mechanism that selects the appropriate voice and pronunciation rules according to the active language.

The system may dynamically switch languages depending on the conversation context or user preferences.

This requires selecting compatible voice models and pronunciation rules for each supported language.

---

### Audio Playback System

#definition: component responsible for playing synthesized audio through the robot's speakers.

This module manages:

* buffering of generated audio
* streaming playback
* synchronization with other outputs
* volume control

It also ensures that speech playback does not conflict with other audio sources such as music or system alerts.

Possible personalization parameters include:

# 7.7 Active Camera

#definition: subsystem responsible for the deliberate and goal-directed use of the robot's camera as an action channel, including image capture, video recording, scanning, streaming, and task-specific visual acquisition.

The Active Camera subsystem manages every case in which NORA intentionally uses its camera to perform an explicit action. This module does not describe passive perception of the environment, but rather the controlled operational use of the camera as an output-capable instrument for achieving a goal.

This distinction is architecturally important.

In the **Vision** module under Perception, the camera is treated as a sensory input source continuously or opportunistically analyzed in order to detect people, faces, objects, gestures, text, or scene context. In contrast, in **Active Camera**, the system intentionally decides to use the camera in order to produce a result, capture evidence, acquire a document, stream visual content, or support another action pipeline.

In other words:

* **Vision (Perception)** answers: "What is the robot seeing?"
* **Active Camera (Action)** answers: "What does the robot intentionally do with the camera?"

Because of this, the module belongs to the **Action and Expression** layer.

---

## Architectural Role

Within the global architecture, Active Camera acts as the execution layer for deliberate camera-related operations.

Commands may originate from several subsystems:

Planner

Selects camera usage when a user goal requires visual acquisition.

Specialized agents

For example, reading, recognition, or productivity agents may request image capture.

Dialogue system

May trigger camera actions in response to user requests such as "take a photo" or "scan this page".

FSM

Constrains when camera actions are allowed depending on operational state.

This subsystem therefore transforms a high-level intent into a concrete camera operation.

---

## Core Responsibilities

The subsystem is responsible for:

* opening and controlling the camera intentionally
* capturing still images
* recording video
* scanning documents
* starting visual streams
* acquiring frames for downstream task pipelines
* storing or forwarding captured media
* synchronizing camera use with privacy, permissions, and system state

This means the subsystem is not only about raw capture, but about **purposeful camera execution with traceable outputs**.

---

## Key Difference from Vision Perception

This distinction must remain explicit in the architecture.

### Vision under Perception

* continuous or event-driven sensing
* detection of people, gestures, objects, and text
* interpretation of the environment
* production of perceptual events

### Active Camera under Action and Expression

* intentional capture requested by a plan or command
* explicit production of files, frames, streams, or scans
* user-visible or system-usable camera operations
* execution governed by permissions and policy

A person standing in front of the robot may be detected by the Vision subsystem without any deliberate camera action being executed. By contrast, when the user says "take a photo of this document," the Active Camera subsystem is engaged because the system is intentionally performing a camera-based action.

---

## Types of Camera Actions

The subsystem may support several categories of deliberate camera usage.

### Photo Capture

The robot can take a still image and store it as a file or pass it to another subsystem.

Typical uses include:

* taking a picture on user request
* capturing an object for later analysis
* saving evidence of a completed task
* attaching an image to a note or report

This is one of the most direct camera actions in the system.

---

### Video Recording

The system may start and stop video recording intentionally.

Typical uses include:

* recording a scene
* capturing a short clip for later review
* documenting an event
* creating media content

Video recording may include duration limits, metadata tagging, and user confirmation requirements.

---

### Document Scanning

The camera may be used as a scanning instrument.

Typical uses include:

* scanning printed pages
* digitizing handwritten notes
* capturing receipts or forms
* preparing an image for OCR

This often involves additional processing such as perspective correction, cropping, enhancement, and document classification.

---

### OCR Acquisition

The subsystem may capture an image specifically for text extraction.

Examples:

* reading a sign
* reading a label
* capturing a page for OCR
* extracting information from a printed document

In this case, Active Camera performs the acquisition, and OCR or Reading modules perform the interpretation.

---

### Recognition Capture

The camera may intentionally capture data for recognition tasks.

Examples:

* capture for face recognition
* capture for object recognition
* capture for barcode or QR reading

Again, the camera action is separated from the recognition engine itself.

---

### Live Stream / Preview

The subsystem may open a live camera stream for a user or another component.

Typical uses include:

* showing live camera preview on the robot screen
* remote monitoring through the frontend
* guiding the user during document positioning
* visual confirmation before taking a photo

Streaming turns the camera into a live interaction channel rather than only a recording device.

1. receive high-level action request

6. optionally forward result to another subsystem

This structured pipeline ensures consistent and auditable camera behavior.

# 7.8 Home Automation / IoT

While other modules in the Action and Expression layer control the robot's internal hardware (such as voice output, movement, or LEDs), this subsystem extends NORA's capabilities beyond the robot body and into the physical environment.

This effectively transforms NORA into an intelligent interface for controlling connected spaces such as homes, offices, or laboratories.
