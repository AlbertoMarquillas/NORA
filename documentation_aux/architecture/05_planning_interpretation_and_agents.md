# 6. Planning, Interpretation and Agents

#definition: architectural layer responsible for transforming interpreted inputs, goals or events into structured plans that the system can execute through tools, agents and actions.

## Overview

The Planning, Interpretation and Agents module is the cognitive decision layer that determines **what NORA should do in response to an input, request, event or conversational objective**.

Previous modules of the architecture perform different roles:

* Perception modules detect signals from the environment.
* The Cognitive Core manages operational state through the FSM and shared system context.
* The Dialogue and Session System manages conversational continuity and long-term interaction context.

Once an input has been detected, interpreted and contextualized, the system must determine how to respond in a structured and goal-oriented way. This responsibility belongs to the Planning, Interpretation and Agents layer.

This module converts high-level intentions into **structured plans composed of steps, tool invocations and agent reasoning processes**.

It therefore acts as the **task intelligence system** of NORA.

## Role in the Global Architecture

Within the overall architecture, this layer acts as a bridge between:

* cognitive understanding
* operational execution

The module receives structured inputs such as:

* interpreted user utterances
* dialogue turns
* detected intentions
* FSM events
* session objectives
* project goals
* external system requests

Using this information, the system constructs a plan that defines:

* what goal should be pursued
* which tools or agents are required
* which sequence of actions should be executed
* what information must be obtained first
* how the result should be communicated to the user

The resulting plan is then executed by downstream modules such as:

* Action and Expression
* External Integrations
* Backend services

## Responsibilities

The Planning, Interpretation and Agents layer is responsible for:

* understanding the semantic meaning of inputs
* determining the user's underlying intention
* mapping requests to system capabilities
* deciding which tools or integrations must be used
* constructing multi-step plans when necessary
* coordinating specialized reasoning modules (agents)
* adapting the plan to the conversational context

The module must operate correctly both for **simple commands** and for **complex multi-step tasks**.

Examples include:

Simple tasks:

* answering a factual question
* playing music
* reading a document
* turning on a light

Complex tasks:

* planning a route
* helping the user study a topic
* solving a mathematical problem
* preparing a recipe
* managing a long conversational learning session

## Types of Inputs Handled

The planning layer can be triggered by several types of inputs across the system.

### Conversational inputs

Examples:

* questions
* commands
* requests
* confirmations
* corrections
* casual conversation

Example:

"Play some jazz music"

### Contextual triggers

Inputs generated from the internal state of the system.

Examples:

* a session goal becoming active
* a reminder being triggered
* a project task requiring continuation

### Event-driven triggers

Events generated by other modules.

Examples:

* EVT_WAKEWORD
* EVT_SPEECH_RECOGNIZED
* EVT_GESTURE_COMMAND
* EVT_SCHEDULE_TRIGGERED

### External system requests

Commands coming from other devices or systems.

Examples:

* frontend request
* API command
* IoT message
* webhook

## Output of the Module

The output of this layer is **not a direct action**, but a structured decision about how to proceed.

Typical outputs include:

* selected intent
* execution plan
* required tools
* selected specialized agent
* required information queries

The plan may include steps such as:

1. request missing information
2. retrieve external data
3. perform computation
4. generate explanation
5. produce multimodal output

The execution of the plan is delegated to the Action layer and other system services.

## Planning Complexity

The module must support different levels of planning complexity.

### Single-step execution

Some requests can be solved directly.

Example:

User: "What is the capital of France?"

Steps:

1. detect knowledge query
2. retrieve answer
3. generate response

### Conditional execution

Some tasks require missing information to be obtained.

Example:

User: "Navigate to a restaurant"

Steps:

1. detect navigation request
2. ask which restaurant
3. wait for user response
4. retrieve route
5. display route

### Multi-step planning

Some tasks require multiple coordinated actions.

Example:

"Plan a study session for tomorrow"

Possible plan:

1. retrieve calendar availability
2. ask preferred study topic
3. allocate time blocks
4. create reminders
5. summarize plan

## Interaction with Other Modules

This module interacts with several architectural components.

### Dialogue and Session System

Provides:

* conversational history
* session goals
* user preferences

### Cognitive Core

Provides:

* current operational state
* active interaction mode
* contextual restrictions

### Tool Integrations

Provides:

* external capabilities
* knowledge sources
* system services

### Action and Expression Layer

Responsible for executing the final outputs produced by the plan.

# 6.1 Semantic Interpretation

#definition: subsystem responsible for transforming raw interpreted inputs (text, gestures, events) into structured semantic representations that can be used by higher‑level reasoning components such as intent detection and planning.

## Overview

Semantic Interpretation is the first reasoning stage within the **Planning, Interpretation and Agents** layer. Its objective is to convert human or system inputs into a structured representation of meaning that the rest of the architecture can reason about.

At this stage, the system does not yet decide *what action to perform*. Instead, it focuses on extracting the semantic structure of the input so that downstream modules can determine the user's intention and plan the correct response.

Inputs reaching this module are already normalized by earlier components in the architecture. For example:

* speech has already been converted to text
* gestures have already been detected
* events have already been classified

Semantic Interpretation therefore operates on **interpretable signals**, not raw sensor data.

Its output is a structured semantic object describing the meaning, structure and context of the input.

## Position in the Architecture

Within the NORA architecture, semantic interpretation sits between perception and intention reasoning.

Pipeline position:

Perception → input normalization → semantic interpretation → intent detection → planning

This separation ensures that:

* perception focuses on sensing
* semantic interpretation focuses on meaning extraction
* intent detection focuses on goal identification

## Goals of the Module

The semantic interpretation subsystem aims to:

* transform natural language into structured semantic representations
* extract relevant entities from inputs
* identify dialogue acts
* resolve contextual references
* enrich the input with session and contextual information
* provide a clean interface for intent detection

By isolating semantic interpretation from intention reasoning, the architecture remains modular and easier to extend.

## Types of Inputs Processed

The module can receive multiple categories of inputs.

### Natural language input

Most user interactions arrive as natural language sentences obtained through speech‑to‑text or typed input.

Examples:

* "Play some music"
* "What is the weather tomorrow?"
* "Help me practice English"
* "Turn on the kitchen lights"

The system must extract the semantic elements contained in these sentences.

### Conversational signals

Some inputs represent conversational responses rather than explicit commands.

Examples:

* "yes"
* "no"
* "stop"
* "repeat"
* "wait"

These correspond to dialogue acts that modify the state of an ongoing interaction.

### Context‑dependent responses

Certain inputs depend heavily on previous conversation turns.

Example:

User: "navigate there"

In this case, the interpreter must resolve the reference "there" using the conversation history or visual context.

### Structured system events

Inputs may also originate from other subsystems.

Examples:

* gesture events
* system commands
* detected visual targets

These signals are converted into semantic representations compatible with the planning pipeline.

## Core Responsibilities

The semantic interpretation module performs several processing tasks.

### Dialogue act classification

The module determines the conversational role of the input.

Possible dialogue acts include:

* question
* command
* request
* confirmation
* rejection
* interruption
* clarification
* conversation

This classification helps downstream modules understand how the input should be handled.

### Entity extraction

Relevant elements contained in the input are extracted as structured entities.

Example:

Input:

"Play jazz music"

Extracted entities:

* action: play
* genre: jazz
* object: music

Another example:

"Turn on the living room lights"

Extracted entities:

* action: turn_on
* device: lights
* location: living_room

### Reference resolution

Natural conversation frequently includes references to previously mentioned objects.

Examples:

* "Play it again"
* "Turn that off"
* "Navigate there"

The interpreter attempts to resolve these references using:

* conversation history
* session context
* perception information

### Language identification

The system identifies the language of the input when multilingual interaction is supported.

This information influences:

* response generation
* speech synthesis
* learning modules

### Context enrichment

The semantic representation is enriched with contextual information from other modules.

Examples of contextual metadata include:

* active user
* session identifier
* current topic
* active project
* previous dialogue state

This contextual layer allows intent detection to reason about the input more effectively.

## Semantic Representation Output

After interpretation, the input is transformed into a structured semantic object.

Example structure:

semantic_input

text: "play jazz music"
language: en
dialogue_act: command
entities:
action: play
genre: jazz
context:
session_id: 42
user_id: 7
confidence: 0.94

Another example:

text: "yes"
dialogue_act: confirmation
target: previous_question
confidence: 0.98

This representation becomes the input to the **Intent Detection module**.

## Ambiguity Handling

Human language frequently contains incomplete or ambiguous instructions.

Example:

"Play music"

Unknown parameters may include:

* genre
* artist
* playlist
* device

When this occurs, the semantic interpreter flags the request as incomplete.

Example output:

intent_candidate: play_music
missing_parameters:

* genre
* playlist

The planner can then request clarification from the user.

## Interaction with Other Modules

The semantic interpretation subsystem interacts with several architectural components.

### Dialogue and Session System

Provides:

* conversation history
* session context
* dialogue state

### Perception modules

Provide signals such as:

* detected gestures
* detected objects
* visual attention

### Cognitive Core

Provides:

* current operational state
* active interaction mode
* contextual restrictions

## Output to Intent Detection

The final output of semantic interpretation is a normalized semantic representation containing:

* normalized input text
* dialogue act classification
* extracted entities
* contextual metadata
* confidence scores

This structured representation is passed to the **Intent Detection module (6.2)**, which determines the goal behind the input.

# 6.2 Intent Detection

#definition: subsystem responsible for identifying the objective or goal behind a semantically interpreted input.

## Overview

Intent Detection is the component responsible for determining **what the user wants to achieve** based on the structured semantic representation produced by the Semantic Interpretation module.

While semantic interpretation extracts meaning and structure from the input, intent detection focuses on identifying the **underlying goal or task** that the user expects the system to perform.

For example, the sentence:

"Play jazz music"

may be semantically interpreted as:

* action: play
* object: music
* genre: jazz

The intent detection module interprets this structure and classifies the request as:

intent: play_music

This intent representation allows the planning system to determine the appropriate sequence of actions required to satisfy the request.

## Role in the Architecture

Intent Detection is the second reasoning stage inside the **Planning, Interpretation and Agents** layer.

Pipeline position:

Perception → Semantic Interpretation → Intent Detection → Planner

This separation ensures that:

* semantic interpretation extracts structured meaning
* intent detection identifies the user's goal
* planning determines how to achieve the goal

## Goals of the Module

The intent detection subsystem aims to:

* identify the objective behind a user input
* classify requests into structured intent categories
* determine whether the request corresponds to a known capability
* detect incomplete requests
* determine when clarification is required
* map user goals to planning strategies

The module therefore acts as the **goal recognition system** of NORA.

## Types of Intent

The system must recognize a wide variety of possible intentions.

### Informational queries

Requests that require retrieving knowledge or information.

Examples:

* "What is the capital of France?"
* "What time is it?"
* "How far is Barcelona?"

Possible intent classification:

intent: knowledge_query

### Action commands

Requests that require the system to perform a concrete action.

Examples:

* "Play music"
* "Turn on the lights"
* "Take a picture"

Possible intent classifications:

* play_music
* control_device
* capture_photo

### Conversational intents

Inputs that represent social or conversational interaction.

Examples:

* "How are you?"
* "Tell me a joke"

Possible intent classification:

intent: casual_conversation

### Learning and tutoring intents

Requests that involve education or training.

Examples:

* "Help me learn English"
* "Explain quantum mechanics"

Possible intent classifications:

* tutoring_session
* explanation_request

### Productivity intents

Requests related to personal organization or productivity.

Examples:

* "Create a reminder"
* "Add this to my calendar"

Possible intent classifications:

* create_reminder
* calendar_event

### Navigation and location intents

Requests involving routes, locations or travel.

Examples:

* "Navigate to the nearest restaurant"

Possible intent classification:

intent: navigation_request

### System configuration intents

Requests that modify system configuration.

Examples:

* "Change language to Spanish"
* "Increase volume"

Possible intent classifications:

* change_language
* adjust_volume

## Context-Aware Intent Detection

Intent detection must consider the **context of the conversation**.

Example:

User: "Play music"

Intent:

play_music

Later in the conversation:

User: "Stop"

The meaning of "stop" depends on the active task. In this context the intent becomes:

stop_music

Contextual reasoning therefore relies on:

* conversation history
* session state
* active tasks

## Handling Ambiguous Inputs

Some inputs may correspond to multiple possible intents.

Example:

"Open it"

Possible interpretations depend on context:

* open_document
* open_application
* open_door

The system resolves ambiguity using:

* conversation context
* visual perception
* recent actions

If ambiguity remains unresolved, the planner may request clarification.

## Intent Representation

The output of the intent detection module is a structured representation of the detected goal.

Example:

intent_object

intent: play_music
confidence: 0.91
entities:
genre: jazz
parameters:
device: default_speaker

Another example:

intent_object

intent: navigation_request
confidence: 0.88
entities:
destination: nearest_restaurant

This object is passed to the **Planner (6.3)**.

## Confidence Scoring

Each detected intent is associated with a confidence score.

This score allows the system to decide whether:

* the intent is reliable
* clarification should be requested
* fallback strategies should be used

Example:

confidence > 0.85 → execute normally

confidence between 0.6 and 0.85 → confirm with user

confidence < 0.6 → ask clarification

Intent Detection interacts with several components of the architecture.

### Semantic Interpretation

### Dialogue and Session System

Provides:

* conversation history
* session goals

### Cognitive Core

Provides:

* system state
* operational mode

### Planner

Receives the detected intent and constructs an execution strategy.

## Output to the Planner

The final output of intent detection includes:

* detected intent
* confidence score
* extracted entities
* contextual parameters

This information is then passed to the **Planner module (6.3)**, which determines how the goal should be achieved.

# 6.3 Planner

#definition: subsystem responsible for constructing an executable strategy that transforms a detected intent into an ordered sequence of steps, decisions, tool invocations and outputs.

## Overview

The Planner is the component that decides **how NORA should achieve a detected goal**.

After the Semantic Interpretation module extracts structured meaning from an input, and the Intent Detection module identifies the underlying user objective, the Planner builds the operational strategy required to satisfy that objective.

This means that the Planner does not focus on understanding *what the input means* and does not directly execute actions either. Instead, its responsibility is to produce a structured plan that describes:

* what must happen next
* in what order the steps should occur
* what information is still missing
* what tools or agents will be needed
* what outputs should be generated
* what conditions or confirmations must be satisfied before execution continues

The Planner is therefore one of the central reasoning components of the NORA architecture. It transforms recognized goals into **coherent, context-aware, and executable workflows**.

## Role in the Architecture

Within the Planning, Interpretation and Agents layer, the Planner receives a detected intent together with its associated semantic structure and contextual metadata, and converts that information into a formal execution plan.

Pipeline position:

Perception → Semantic Interpretation → Intent Detection → Planner → Tool Selector → Specialized Agents → Action and Expression

This separation preserves architectural clarity:

* Semantic Interpretation extracts structured meaning
* Intent Detection identifies the user goal
* Planner designs the execution strategy
* Tool Selector decides which integrations or internal capabilities must be used
* Specialized Agents solve domain-specific reasoning problems
* Action and Expression executes the final outputs

## Goals of the Module

The Planner subsystem aims to:

* transform intents into executable plans
* define the sequence of steps required to complete a task
* determine whether information is missing
* decide when clarification or confirmation is needed
* choose between direct execution and multi-step execution
* coordinate downstream reasoning and execution modules
* adapt the plan to the active context, user profile and system state
* handle interruptions, cancellations and replanning

The Planner therefore acts as the **task orchestration brain** of the architecture.

## Why Planning is Necessary

Not all user inputs can be executed immediately.

Some requests are complete and can be solved in a single step.

Example:

"What time is it?"

Possible plan:

1. retrieve current time
2. generate response
3. speak and display answer

However, many requests are incomplete, ambiguous, conditional or multi-step.

Example:

"Navigate to a restaurant"

This request requires additional decisions:

* which restaurant
* from which starting point
* whether to show the route on screen
* whether to use walking or driving mode

Possible plan:

1. detect missing destination details
2. ask clarification question
3. wait for user response
4. retrieve route
5. summarize route
6. display route
7. speak result

Without a Planner, the system would only react to individual commands. With a Planner, NORA can behave as a coherent assistant capable of structured task completion.

## Types of Planning

The Planner must support several levels of planning complexity.

### Single-step planning

A direct request can sometimes be satisfied with a simple plan.

Example:

User input: "Increase volume"

Plan:

1. validate permission and current state
2. adjust output volume
3. confirm change

### Conditional planning

Some tasks depend on missing parameters, permissions or user confirmation.

Example:

User input: "Delete this session"

Plan:

1. identify target session
2. request confirmation because action is destructive
3. wait for confirmation
4. delete session
5. report result

### Multi-step planning

Some requests require several coordinated actions across different subsystems.

Example:

User input: "Plan my study session for tomorrow"

Possible plan:

1. retrieve calendar availability
2. identify active study project or ask for topic
3. estimate available time blocks
4. propose a study schedule
5. ask for confirmation
6. create reminders
7. summarize the final plan

### Reactive planning

The system may need to react to internal events or environmental conditions rather than direct user requests.

Example:

Trigger: low battery detected during active interaction

Possible plan:

1. assess whether current task can be safely interrupted
2. inform user if interaction is active
3. save session state if required
4. enter safe power-saving mode or return to charging workflow

### Replanning

The Planner must also support plan modification when new information arrives or conditions change.

Example:

* user changes destination
* internet connection fails
* selected tool becomes unavailable
* conversation is interrupted by a higher-priority event

In such cases, the Planner must rebuild or adapt the execution strategy.

## Inputs to the Planner

The Planner consumes information from multiple modules.

### From Intent Detection

* detected intent
* confidence score
* extracted parameters
* missing parameters

### From Semantic Interpretation

* dialogue act
* entities
* reference resolution results
* semantic structure

### From Dialogue and Session System

* active session
* conversation history
* project context
* remembered decisions
* tone or interaction preferences

### From Cognitive Core

* current FSM state
* interaction mode
* current user
* active restrictions
* operational health of modules

### From User Profile and Permissions

* allowed capabilities
* personalization preferences
* device access rights
* configured defaults

## Core Responsibilities

The Planner performs a series of decision-making responsibilities.

### Goal decomposition

A high-level goal may need to be divided into smaller operational steps.

Example:

Goal: "Help me cook pasta"

Possible decomposition:

* identify recipe type
* retrieve recipe instructions
* present steps incrementally
* answer follow-up questions
* optionally set timers

### Missing-information detection

The Planner determines whether the current request contains enough information for execution.

Example:

Input: "Send an email"

Possible missing fields:

* recipient
* subject
* message content

If required data is missing, the plan includes clarification steps.

### Confirmation policy

Certain actions should not be executed immediately.

Examples:

* deleting data
* opening a door
* controlling home devices
* sending messages to external people
* making purchases in future extensions

The Planner defines whether a confirmation step is mandatory before execution.

### Step ordering

The Planner determines the correct sequence of actions.

Example:

For "Read this document and summarize it":

1. capture or retrieve document
2. extract text
3. analyze content
4. generate summary
5. present result

### Branching and conditions

Plans may contain branches.

Example:

If weather API is available:

* query weather service

Else:

* report that real-time weather data is unavailable

The Planner must encode these conditional paths.

### Failure-aware design

The Planner must consider what happens if a step fails.

Examples:

* tool timeout
* authentication failure
* missing camera stream
* route search unavailable

Instead of assuming ideal execution, the plan should include fallback behavior whenever possible.

## Plan Structure

The output of the Planner should be a structured plan object, not an informal description.

A plan may contain:

* plan identifier
* originating intent
* objective description
* ordered steps
* dependencies between steps
* required tools
* required agents
* preconditions
* clarification requirements
* confirmation requirements
* fallback rules
* expected output modalities
* execution priority

Example conceptual structure:

plan_object

plan_id: p_1042
intent: navigation_request
objective: guide the user to a selected restaurant
status: ready_for_clarification
missing_information:

* destination
  steps:
* ask_user_for_destination
* retrieve_route
* summarize_route
* display_map
* speak_result
  outputs:
* screen
* voice
  priority: normal

Another example:

plan_object

plan_id: p_2018
intent: knowledge_query
objective: answer factual question
status: ready
steps:

* retrieve_information
* generate_answer
* speak_answer

## Planning Policies

The Planner should follow consistent architectural policies.

### Minimum necessary execution

The system should avoid unnecessary steps.

If a request can be solved directly, the plan should remain simple.

### Clarify only when needed

The system should ask follow-up questions only when necessary to avoid excessive friction.

### Use context before asking

If missing information can be reliably inferred from context, previous turns or user defaults, the Planner may avoid clarification.

Example:

User: "Play music"

If the user has a default music provider and preferred speaker, the plan can use those defaults.

### Prioritize safety and permissions

Potentially sensitive actions must respect permission policies and system restrictions.

### Remain interruptible

Long or multi-step plans should be interruptible, cancellable and resumable.

## Interaction with the FSM and Operational State

Planning cannot be isolated from the operational condition of the robot.

Examples:

* if NORA is already speaking, the plan may need to queue or delay a spoken response
* if the camera is unavailable, visual actions cannot be planned normally
* if the system is in admin mode, some user-facing actions may be disabled
* if the user is unauthenticated, privileged plans may be rejected or downgraded

The Planner must therefore generate plans that are **state-aware** and **capability-aware**.

## Interaction with Tool Selector and Agents

The Planner decides **what kinds of capabilities are needed**, but not always the exact implementation.

For example, the Planner may decide that the following are required:

* map lookup
* weather retrieval
* OCR extraction
* multimedia playback
* document summarization

The detailed selection of which tool, engine or provider to use belongs to the **Tool Selector (6.4)**.

Likewise, if the request belongs to a complex domain such as tutoring, navigation or productivity, the Planner can delegate reasoning to a **Specialized Agent (6.5)**.

In this sense:

* Planner defines the workflow
* Tool Selector chooses the execution instruments
* Specialized Agents perform domain-specific reasoning

## Handling Interruptions and Cancellations

A robust planner must support live interaction changes.

Examples:

* user says "stop"
* user changes their mind
* a higher-priority system event occurs
* network connectivity is lost mid-task

The Planner should support:

* cancellation of pending steps
* pausing long workflows
* resuming suspended workflows
* replacing obsolete subplans

This is especially important in embodied systems where perception, dialogue and action happen continuously.

## Example Planning Scenarios

### Example 1: simple factual question

User: "What is the weather today?"

Possible plan:

1. check whether location is known
2. retrieve weather data
3. generate concise answer
4. speak answer
5. optionally display answer on screen

### Example 2: multimodal reading request

User: "Read this paper and summarize it"

Possible plan:

1. identify document source
2. capture or retrieve the document
3. extract text or parse file
4. send content to reading/summarization capability
5. generate summary
6. speak summary or display it

### Example 3: device control with confirmation

User: "Open the front door"

Possible plan:

1. verify authentication and permissions
2. verify that the target device is available
3. ask for confirmation if policy requires it
4. execute device command
5. log sensitive action
6. report result

### Example 4: tutoring interaction

User: "Help me practice English"

Possible plan:

1. identify active learning session or create one
2. retrieve user language level and preferences
3. select learning mode
4. start tutoring dialogue
5. adapt responses to user performance
6. save key progress into session memory

## Output to the Next Modules

The final output of the Planner is a structured execution strategy describing:

* what must be done
* in what sequence
* under what conditions
* with which required capabilities
* with what expected outputs

This plan is then passed downstream to:

* the Tool Selector, which chooses concrete integrations or engines
* Specialized Agents, when domain-specific reasoning is needed
* the Action and Expression layer, once executable outputs are ready

## Architectural Importance

The Planner is one of the most important subsystems in the entire NORA architecture.

Without planning, NORA would behave like a collection of disconnected reactions. With a Planner, the system can behave as a coherent assistant capable of:

* multi-step task completion
* contextual follow-up
* clarification when needed
* safe execution of sensitive actions
* adaptation to changing conditions
* coordination across dialogue, tools, memory and physical actions

For this reason, the Planner should be treated as a first-class architectural component, tightly integrated with context, state, permissions and downstream execution systems.

# 6.4 Tool Selector

#definition: subsystem responsible for selecting the most appropriate tool, integration, service, engine or internal capability required to execute each step of a plan.

## Overview

The Tool Selector is the component that decides **with what concrete capability NORA should execute a planned operation**.

After the Planner defines the execution strategy, the system still needs to determine which actual tool, engine, provider, integration or internal module should be used for each step. This is the responsibility of the Tool Selector.

For example, the Planner may decide that a task requires:

* web information retrieval
* route lookup
* OCR processing
* document summarization
* music playback
* smart home control

However, the Planner should not hardcode exactly which implementation must be used. Instead, it expresses the required capability. The Tool Selector then resolves that capability into a concrete execution option.

This makes the architecture modular, replaceable and adaptable. The same high-level plan can be executed through different tools depending on availability, permissions, latency, cost, privacy policy, hardware state or user preferences.

The Tool Selector therefore acts as the **capability resolution layer** of NORA.

## Role in the Architecture

Within the Planning, Interpretation and Agents layer, the Tool Selector sits after the Planner and before specialized execution or action.

Pipeline position:

Perception → Semantic Interpretation → Intent Detection → Planner → Tool Selector → Specialized Agents / Action and Expression / External Integrations

This separation preserves clear architectural roles:

* Semantic Interpretation extracts meaning
* Intent Detection identifies the goal
* Planner decides what must be done
* Tool Selector decides what concrete tool should do it
* Specialized Agents reason within a domain
* Action and Expression executes outputs into the world

## Goals of the Module

The Tool Selector subsystem aims to:

* map abstract plan requirements to concrete tools
* choose among multiple equivalent providers
* account for system state, permissions and policies
* account for user preferences and defaults
* minimize unnecessary cost, latency or privacy exposure
* detect when no valid tool is available
* support fallback selection when the preferred tool fails
* keep tool usage decoupled from high-level planning

This module ensures that the rest of the architecture can remain capability-oriented instead of implementation-oriented.

## Why Tool Selection is Necessary

A single user goal may be satisfiable through multiple alternative mechanisms.

Examples:

* a knowledge request may be answered using memory, local retrieval, web search or an external knowledge source
* a navigation request may use a maps API, a local route engine or a cached location service
* a music request may use Spotify, YouTube or a local media library
* OCR may be performed by a local engine or a cloud provider
* text generation may use a local LLM or a remote model provider

Without an explicit Tool Selector, the system would either:

* hardcode one provider everywhere
* duplicate decision logic across modules
* become difficult to maintain and extend

By introducing this subsystem, NORA can separate:

* **what capability is required**
* **which implementation should be used**

## Core Responsibilities

The Tool Selector performs several important responsibilities.

### Capability resolution

The Planner may specify that a step requires a generic capability such as:

* retrieve_weather
* search_web
* play_media
* control_device
* read_document

The Tool Selector resolves each capability into a specific implementation.

Example:

Capability: play_media

Possible resolution:

* Spotify
* YouTube
* local player

### Provider choice

When multiple providers are available, the Tool Selector chooses the most appropriate one.

Selection criteria may include:

* user default provider
* current provider availability
* authentication state
* privacy requirements
* execution speed
* response quality
* monetary cost in future deployments

### Pre-execution validation

Before selecting a tool, the subsystem may validate whether the tool can actually be used.

Examples:

* user is authenticated with Spotify
* the camera is currently operational
* internet connectivity is available
* the calendar integration is connected
* the user has permission to use the admin tool

### Fallback selection

If the preferred tool is unavailable, the Tool Selector may choose an alternative.

Example:

Preferred provider: Spotify
Fallback provider: YouTube
Second fallback: local music library

### Policy-aware selection

Some tools may be disallowed depending on context.

Examples:

* external web search disabled in privacy mode
* email sending restricted for guest users
* door unlocking unavailable without biometric confirmation
* cloud OCR disabled for confidential documents

### Multi-tool composition

Certain tasks may require more than one tool.

Example:

Task: "Read this document and summarize it"

Possible selected chain:

1. document retrieval tool
2. OCR or parser
3. summarization tool
4. TTS output tool

The Tool Selector must support coordinated selection across multiple plan steps.

## Selection Criteria

Tool selection should be based on explicit architectural criteria.

### Functional compatibility

The tool must satisfy the required capability.

### Availability

The tool must currently be reachable and operational.

### Permission compatibility

The action must be allowed for the active user and mode.

### Context suitability

The selected tool must make sense in the current context.

Example:

If the request is offline-capable and privacy-sensitive, a local tool may be preferred over a cloud service.

### User preference alignment

If the user has configured a preferred provider, it should be respected when possible.

### Performance and efficiency

Latency, reliability and cost should be considered when relevant.

### Safety and privacy

Sensitive operations should prefer safer or more private execution paths when possible.

## Tool Registry Concept

Architecturally, tool selection is easier to maintain if the system exposes a **tool registry** or capability catalog.

Each registered tool may declare metadata such as:

* tool identifier
* capability type
* supported inputs
* supported outputs
* required permissions
* online/offline requirement
* privacy level
* authentication requirement
* cost class
* priority score
* fallback rank
* health status

Example conceptual registry entry:

tool_id: spotify_playback
capability: play_media
requires_auth: true
supports:

* music
* playlists
  privacy_level: external_service
  priority: high
  fallback_rank: 1

This registry-based design avoids scattering selection logic across the codebase.

## Architectural Importance

The Tool Selector is essential for making NORA modular, extensible and robust.

Without it, planning logic would become tightly coupled to specific implementations, and replacing providers or adapting to runtime conditions would be much harder.

By introducing a dedicated tool selection layer, NORA gains:

* easier integration of new providers
* provider fallback support
* policy-aware execution
* user-preference-aware execution
* runtime adaptability

For this reason, the Tool Selector should be treated as a core subsystem of the planning architecture rather than as a minor helper component.

# 6.5 Specialized Agents

#definition: modules with domain-specific reasoning capabilities that can execute complex tasks using planning, memory, tools and dialogue within a particular problem domain.

## Overview

Specialized Agents are reasoning components dedicated to handling **specific categories of tasks that require domain knowledge, structured workflows, or extended interaction**.

While the Planner defines the global execution strategy and the Tool Selector resolves which capabilities should be used, Specialized Agents provide **deep reasoning within a particular domain**.

Instead of treating every request as a generic operation, the architecture allows some problems to be delegated to agents that understand the structure of a domain such as:

* conversation
* learning
* knowledge retrieval
* navigation
* productivity
* multimedia
* home automation
* document analysis
* administration

Each agent encapsulates the logic, tools, and strategies required to operate effectively in that domain.

This approach improves:

* modularity
* reasoning quality
* maintainability
* extensibility

Agents can be added, replaced or improved without changing the rest of the system architecture.

## Role in the Architecture

Specialized Agents operate downstream from the Planner and Tool Selector.

Pipeline position:

Perception → Semantic Interpretation → Intent Detection → Planner → Tool Selector → Specialized Agents → Action and Expression

The Planner determines **what must be achieved**, while the Specialized Agent determines **how to reason within its domain in order to achieve that goal**.

In many cases the Planner may delegate an entire task to a specialized agent rather than executing each step directly.

Example:

User request:

"Help me practice English"

Planner decision:

* delegate to Learning Agent

The Learning Agent then manages the interaction using its own domain-specific logic.

## Goals of the Module

The Specialized Agents subsystem aims to:

* encapsulate domain-specific reasoning
* manage complex workflows inside a domain
* coordinate tool usage within the domain
* maintain domain-specific context
* interact conversationally when required
* adapt behavior based on user interaction

This allows NORA to scale across many capabilities without creating a single monolithic reasoning system.

## Agent Responsibilities

A specialized agent may perform several types of tasks.

### Domain reasoning

The agent understands the logic of its domain.

Example:

A Navigation Agent understands concepts such as:

* route
* origin
* destination
* transportation mode
* distance
* estimated time

A Learning Agent understands:

* skill level
* exercises
* explanations
* feedback
* progress tracking

### Task decomposition

Within its domain, the agent may decompose a request into smaller steps.

Example:

User request:

"Explain quantum mechanics simply"

The Knowledge Agent might:

1. determine the user's level
2. retrieve relevant explanations
3. simplify technical language
4. provide examples
5. check understanding

### Tool orchestration

Agents may invoke tools selected by the Tool Selector.

Example:

A Reading Agent may use:

* document loader
* OCR engine
* summarization model
* TTS engine

The agent coordinates these tools to achieve the task.

### Dialogue management inside the domain

Some tasks require ongoing conversation.

Example:

A Learning Agent conducting a tutoring session may:

* ask questions
* evaluate responses
* provide feedback
* adjust difficulty

This dialogue is domain-specific rather than generic conversation.

### State management

Agents may maintain internal state during a task.

Example:

A Productivity Agent managing reminders may track:

* created reminders
* pending confirmations
* recurring schedules

This state is separate from global system state.

## Agent Invocation

The Planner determines when a specialized agent should be used.

Typical invocation cases include:

* complex domain reasoning
* long-running workflows
* structured tutoring or learning
* navigation assistance
* document analysis
* productivity workflows
