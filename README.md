# NORA

## Overview

NORA is a modular cognitive system architecture designed to operate as an embodied, multimodal, persistent, and extensible intelligent platform.

The system integrates perception, dialogue, reasoning, planning, action, memory, and external integrations into a coherent architecture capable of interacting with humans, digital environments, and physical devices.

Rather than being a single application, NORA is designed as a structured software ecosystem composed of multiple architectural domains that collaborate through clearly defined boundaries.

The architecture emphasizes:

* modularity
* strong separation of concerns
* multimodal interaction
* extensibility through agents and tools
* hardware awareness
* persistent contextual memory
* operational observability

The repository structure mirrors the conceptual architecture so that implementation remains aligned with system design.

## Architectural Domains

The system is organized into several major architectural modules.

### Identity, Access and Security

Defines identities, authentication mechanisms, authorization policies, user roles, and system protection mechanisms.

### Perception of the Environment

### Cognitive Core

Maintains internal operational state, behavioural context, and system-level control memory required for runtime decision processes.

### Dialogue and Session System

Manages conversational continuity through sessions, dialogue turns, contextual memory, and project-level conversational context.

### Planning, Interpretation and Agents

Transforms interpreted inputs and contextual goals into structured execution strategies composed of plans, tools, and specialized agents capable of solving domain-specific tasks. fileciteturn2file11

### Action and Expression

Implements the mechanisms through which NORA produces observable outputs and executes actions in digital or physical environments.

### Backend and Application

Provides the operational coordination layer of the system, exposing APIs, orchestrating use cases, routing events, and distributing runtime state across connected clients. fileciteturn2file10

### Persistence and Memory

Provides durable storage for structured records, contextual memory, artifacts, and operational history.

### Frontend and Visualization

Defines the visual interaction environments through which users and operators observe, control, and interact with the system.

### Infrastructure and Hardware

Defines the physical and computational substrate supporting the system, including compute nodes, sensors, actuators, and connectivity structures.

### Integrations, Engines and External Services

Defines the interfaces through which NORA connects to external platforms, engines, APIs, and services.

## Repository Structure

The repository is organized as a multi-layer engineering project designed to separate implementation concerns and maintain scalability.

```
nora/
├── docs/
├── config/
├── scripts/
├── deployment/
├── data/
├── frontend/
├── mobile/
├── firmware/
├── tests/
└── src/
```

### docs/

Contains the complete documentation of the system including architectural specifications, diagrams, API documentation, hardware documentation, deployment documentation, and architectural decision records.

### config/

Contains non-code configuration used to control runtime behaviour including application settings, logging configuration, hardware configuration, and environment-specific settings.

### scripts/

Contains operational scripts used to start services, run migrations, seed initial data, perform diagnostics, or execute maintenance tasks.

### deployment/

Contains infrastructure configuration required to deploy the system in different environments, including container definitions, service supervision configuration, and orchestration manifests.

### data/

Stores runtime data generated by the system including artifacts, caches, uploads, and exported outputs.

### frontend/

Contains the web frontend application responsible for the browser-based user interface.

### mobile/

Contains the mobile application implementation when present.

### firmware/

Contains embedded software for hardware controllers and edge devices.

### tests/

Contains the complete validation architecture including unit tests, integration tests, contract tests, end-to-end tests, performance tests, and hardware-in-the-loop tests.

### src/

Contains the primary implementation of the NORA system.

The codebase follows an architecture-first organization where each directory corresponds to a major architectural domain.

```
src/nora/
├── bootstrap
├── shared
├── identity_access_security
├── interaction_interfaces
├── perception
├── cognitive_core
├── dialogue_and_session
├── planning_and_agents
├── action_and_expression
├── persistence_and_memory
├── backend_and_application
├── frontend_support
├── integrations_and_external_services
├── infrastructure_and_hardware
└── observability
```

This structure ensures a direct mapping between architecture documents and implementation modules.

## Architectural Principles

The system follows several design principles intended to preserve long-term maintainability and clarity.

### Architecture-first organization

The repository structure mirrors the conceptual architecture of the system so that design and implementation remain aligned.

### Strong separation of concerns

Each module is responsible for a clearly defined domain responsibility and avoids mixing unrelated logic.

### Domain isolation

Architectural modules evolve independently with minimal accidental coupling.

### Small explicit files

The project favors many small files with single responsibilities rather than large files with mixed logic.

### Explicit runtime boundaries

Configuration, source code, deployment assets, documentation, and runtime data are separated into distinct repository areas.

### Naming conventions as architectural constraints

File naming follows strict conventions in order to preserve architectural clarity.

Examples include:

* `*_enum.py` for enumerations
* `*_repository.py` for persistence repositories
* `*_provider.py` for capability providers
* `*_adapter.py` for external technology bindings
* `*_policy.py` for rule evaluation components
* `verb_object_service.py` for services

Generic filenames such as `utils.py`, `helpers.py`, or `manager.py` are intentionally avoided to prevent architectural drift.

## Development Setup

Basic development setup typically involves:

1. cloning the repository
2. installing Python dependencies
3. configuring environment variables
4. loading development configuration
5. starting backend services

Example:

```
git clone <repository_url>
cd nora
pip install -r requirements.txt
cp .env.example .env
```

System startup scripts are located in the `scripts/` directory.

## System Execution

Typical runtime components include:

* backend application server
* background workers
* scheduler processes
* realtime communication gateway

Startup utilities are provided through repository scripts to maintain explicit operational workflows.
